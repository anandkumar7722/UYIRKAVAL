# SHE-SHIELD вЂ“ Field Origin Reference

> **Purpose:** This document defines, for every API endpoint and every field inside it,
> whether the value is produced by the **Android Frontend**, the **FastAPI Backend**,
> **Supabase (database layer)**, or the **user themselves**. It also explains exactly
> *how* each value is produced.
>
> Read this before integrating the app, adding new fields, or debugging authentication
> failures.

## Pre-Requisite: Registration Fields (Stored Locally + in `profiles` Table)

Before any endpoint can be called, these values must exist. They are created **once** during
sign-up and stored both in Supabase and in `EncryptedSharedPreferences` on the device.

| Field | Origin | How It Is Generated |
|-------|:------:|---------------------|
| `victim_id` (UUID) | рџ—„пёЏ Supabase Auth | Auto-generated by Supabase's `auth.users` table when the user creates an account (`supabase.auth.signUp()`). Returned to the app as `user.id`. |
| `emergency_token` (string в‰Ґ 8 chars) | рџ“± Android Frontend | Generated on the device at registration time using `java.security.SecureRandom` (12 random bytes в†’ Base64-URL encoded в†’ ~16-character opaque string). Saved to `profiles.emergency_token` and stored in `EncryptedSharedPreferences`. **Never re-generated unless the user explicitly resets it**. |
| `secure_pin` (string в‰Ґ 4 chars) | рџ‘¤ User input | A PIN chosen by the user in the Settings screen. Saved to `profiles.secure_pin`. Used **only** for resolving an SOS вЂ“ never for triggering it. |

## Endpoint 1 вЂ“ `POST /api/sos/relay`

**Purpose:** A victim with **no internet** broadcasts an SOS over Bluetooth mesh (Bridgefy).
A nearby device with internet receives the `SOSPacket` and forwards it here.

### Who Sends This Request?

The **relay device** (рџ”Ѓ) builds the HTTP body by translating the `SOSPacket` it received
over Bluetooth from the victim's phone. The field *values* originate from the victim's
device; only the HTTP transport is performed by the relay.

### Request Fields

| Field | Origin | How It Is Generated |
|-------|:------:|---------------------|
| `victim_id` | рџ“± Victim's device в†’ рџ”Ѓ Relay | The victim's `profiles.id` UUID. Stored locally since registration (`EncryptedSharedPreferences`). Embedded in the `SOSPacket.userUUID` field, which is broadcast over Bluetooth. The relay extracts it from the packet and places it here. |
| `emergency_token` | рџ“± Victim's device в†’ рџ”Ѓ Relay | The token generated at registration (see Pre-Requisite). Embedded in the Bluetooth broadcast packet. The relay forwards it verbatim. The backend then verifies this against `profiles.emergency_token`. |
| `lat` | рџ“± Victim's device в†’ рџ”Ѓ Relay | Latitude captured at the moment the SOS was triggered via `LocationHelper.getLatLng()`, which uses Android's `FusedLocationProviderClient` (`Priority.PRIORITY_HIGH_ACCURACY`, updated every 5 s / 5 m). Stored as `SOSPacket.lat` and broadcast over Bluetooth. |
| `lng` | рџ“± Victim's device в†’ рџ”Ѓ Relay | Longitude, same source as `lat`. |
| `trigger_method` | рџ“± Victim's device в†’ рџ”Ѓ Relay | A string set at trigger time by the victim's app depending on what caused the SOS: `"button"` (manual tap), `"shake"` (FallDetectionService free-fall + impact model), `"voice"` (ScreamDetectionService / YAMNet), `"auto"` (AI auto-detect). Embedded in `SOSPacket.emergencyType` and forwarded by the relay. |
| `risk_score` *(optional)* | рџ“± Victim's device в†’ рџ”Ѓ Relay | On-device ML score (0вЂ“100). For **scream** triggers: `ScreamDetectionService` runs YAMNet TFLite inference; the top-class confidence (0.0вЂ“1.0) Г— 100 becomes the score. For **fall** triggers: `FallDetectionService` derives a score from the impact G-force magnitude. Omit or send `null` when unavailable (e.g. manual button press). |
| `battery_level` *(optional)* | рџ“± Victim's device в†’ рџ”Ѓ Relay | Read from Android's `BatteryManager.EXTRA_LEVEL` / `EXTRA_SCALE` on the victim's device at trigger time. If unavailable, omit or send `null`. |

### Backend-Generated Fields (NOT in request body)

These fields are created by the server during processing and are **never** sent by the client:

| Field | Origin | How It Is Generated |
|-------|:------:|---------------------|
| `relay_ip` | рџ–ҐпёЏ Backend | Extracted from the HTTP request by `_get_client_ip()`. Reads `X-Forwarded-For` header first (for reverse-proxy deployments), falls back to `request.client.host`. Stored in `sos_events.relay_ip`. |
| `status = "active"` | рџ–ҐпёЏ Backend | Hardcoded in `_process_sos()` at insert time. |
| `sos_id` (UUID) | рџ—„пёЏ Supabase | Auto-generated primary key UUID by PostgreSQL when a row is inserted into `sos_events`. Returned to the caller in the response. |
| `created_at` | рџ—„пёЏ Supabase | `DEFAULT now()` PostgreSQL column. Set at insert time by the database. |
| Idempotency check | рџ–ҐпёЏ Backend | `_check_active_sos_within(victim_id, minutes=2)` queries `sos_events` for any `status="active"` row created in the last 2 minutes for this `victim_id`. If found, the existing `sos_id` is returned and no new row is written. |
| Initial `live_tracking` row | рџ–ҐпёЏ Backend | After inserting into `sos_events`, `_process_sos()` writes an initial tracking row using `lat`, `lng`, `battery_level` from the request and `updated_at = datetime.now(UTC)`. |
| SMS background task | рџ–ҐпёЏ Backend | `_send_sos_sms()` is enqueued as a `BackgroundTask`. It fetches trusted contacts from `trusted_contacts` (where `notify_via_sms = true`) and calls Fast2SMS bulk API with the tracking URL. |

### Success Response Fields

| Field | Origin | Description |
|-------|:------:|-------------|
| `success` | рџ–ҐпёЏ Backend | Always `true` on a 201. |
| `message` | рџ–ҐпёЏ Backend | `"SOS event created and contacts are being notified."` (new) or `"Active SOS already exists (idempotent)."` (duplicate). |
| `sos_id` | рџ—„пёЏ Supabase в†’ рџ–ҐпёЏ Backend | The UUID of the newly created (or existing) SOS event. The app **must store this locally** for use in `/location` and `/resolve` calls. |
| `is_new` | рџ–ҐпёЏ Backend | `true` if a fresh SOS row was inserted; `false` if an idempotent hit returned an existing active SOS. |

## Endpoint 2 вЂ“ `POST /api/sos/trigger`

**Purpose:** The victim's phone **has internet** and sends the SOS directly, bypassing
the Bluetooth mesh relay.

### Who Sends This Request?

The **victim's own device** (рџ“±) sends this request directly to the backend.

### Request Fields

| Field | Origin | How It Is Generated |
|-------|:------:|---------------------|
| `victim_id` | рџ“± Victim's device | Loaded from `EncryptedSharedPreferences` where it was stored after Supabase sign-up. |
| `emergency_token` | рџ“± Victim's device | Loaded from `EncryptedSharedPreferences` (set at registration). |
| `lat` | рџ“± Victim's device | `LocationHelper.getLatLng()` via `FusedLocationProviderClient`. Updated every 5 s / 5 m minimum displacement. At trigger time the most recent cached location is used. |
| `lng` | рџ“± Victim's device | Same source as `lat`. |
| `trigger_method` | рџ“± Victim's device | Determined by which code path fired the SOS: `"button"` (manual tap in `HomeScreen` or `SosCountdownScreen`), `"shake"` (FallDetectionService), `"voice"` (ScreamDetectionService), `"auto"` (AI auto-detect). Set programmatically at the call site. |
| `risk_score` *(optional)* | рџ“± Victim's device | On-device ML score (0вЂ“100). Computed by `ScreamDetectionService` (YAMNet confidence Г— 100) or `FallDetectionService` (G-force derived). `null` for manual button triggers. |
| `battery_level` *(optional)* | рџ“± Victim's device | `BatteryManager.EXTRA_LEVEL * 100 / BatteryManager.EXTRA_SCALE`. `null` if `BatteryManager` is unavailable. |

### Backend-Generated Fields (NOT in request body)

Identical to `/api/sos/relay` except `relay_ip` is the **victim's own IP** (not a third-party relay).

| Field | Origin | How It Is Generated |
|-------|:------:|---------------------|
| `relay_ip` | рџ–ҐпёЏ Backend | `_get_client_ip(request)` вЂ” in this case the caller is the victim's device directly. |
| `status = "active"` | рџ–ҐпёЏ Backend | Hardcoded at insert. |
| `sos_id` | рџ—„пёЏ Supabase | PostgreSQL auto-UUID on insert into `sos_events`. |
| `created_at` | рџ—„пёЏ Supabase | `DEFAULT now()`. |
| Idempotency check | рџ–ҐпёЏ Backend | Same 2-minute window check as `/relay`. |
| Initial `live_tracking` row | рџ–ҐпёЏ Backend | Same as `/relay` вЂ” `lat`, `lng`, `battery_level` + `updated_at = now(UTC)`. |
| SMS background task | рџ–ҐпёЏ Backend | Same `_send_sos_sms()` background task. |

### Success Response Fields

Identical to `/api/sos/relay` вЂ” see above.

## Endpoint 3 вЂ“ `POST /api/sos/location`

**Purpose:** Called **periodically** (~every 5вЂ“10 s) by the victim's device while an SOS
is active. Upserts real-time GPS telemetry into `live_tracking` keyed on `sos_id`.

### Who Sends This Request?

The **victim's own device** (рџ“±) always sends this directly. There is no mesh relay
for live tracking (the victim must have internet to send periodic updates).

### Request Fields

| Field | Origin | How It Is Generated |
|-------|:------:|---------------------|
| `sos_id` | рџ“± Victim's device | Returned by `/relay` or `/trigger` and **stored locally** (e.g., `EncryptedSharedPreferences`) immediately after SOS creation. This is the only way the app learns the `sos_id`. |
| `victim_id` | рџ“± Victim's device | Loaded from `EncryptedSharedPreferences` (set at registration). Required here so the backend can verify the caller owns this SOS. |
| `emergency_token` | рџ“± Victim's device | Loaded from `EncryptedSharedPreferences`. Passed to `_verify_emergency_token()` on the server. |
| `lat` | рџ“± Victim's device | Fresh GPS reading from `LocationHelper.getLatLng()` (`FusedLocationProviderClient`). This is a **live** position, not the original SOS trigger location. |
| `lng` | рџ“± Victim's device | Same source as `lat`. |
| `accuracy` *(optional)* | рџ“± Victim's device | `Location.accuracy` from Android's `FusedLocationProviderClient` вЂ” estimated horizontal accuracy radius in metres (1-sigma). Available when GPS has a good fix. |
| `speed` *(optional)* | рџ“± Victim's device | `Location.speed` вЂ” ground speed in m/s computed by the device's location provider. Non-zero only when the device is moving enough for the sensor to detect velocity. |
| `heading` *(optional)* | рџ“± Victim's device | `Location.bearing` вЂ” compass bearing in degrees (0 = North, 90 = East, clockwise). Only valid when `Location.hasBearing()` returns `true`. |
| `battery_level` *(optional)* | рџ“± Victim's device | Current reading from `BatteryManager` at the time of *this* location update (may have changed since SOS creation). |

### Backend-Generated Fields (NOT in request body)

| Field | Origin | How It Is Generated |
|-------|:------:|---------------------|
| `updated_at` | рџ–ҐпёЏ Backend | `datetime.now(timezone.utc).isoformat()` вЂ” set server-side in `_process_sos()` and location upsert. Ensures the timestamp is trustworthy (not spoofable by the client). |
| Ownership + active SOS guard | рџ–ҐпёЏ Backend | After auth, the backend queries `sos_events` to confirm that the given `sos_id` belongs to this `victim_id` AND has `status = "active"`. Returns 404 if not. |
| Upsert logic | рџ–ҐпёЏ Backend | `supabase.table("live_tracking").upsert(..., on_conflict="sos_id")` вЂ” only the most recent location per SOS is kept in the table. |

### Success Response Fields

| Field | Origin | Description |
|-------|:------:|-------------|
| `success` | рџ–ҐпёЏ Backend | Always `true` on 200. |
| `message` | рџ–ҐпёЏ Backend | `"Location updated."` |
| `sos_id` | рџ–ҐпёЏ Backend | Echo of the `sos_id` from the request (for client-side confirmation). |

## Endpoint 4 вЂ“ `POST /api/sos/resolve`

**Purpose:** The victim confirms they are safe. Sets the SOS status to `"resolved"` and
notifies trusted contacts via SMS.

### Who Sends This Request?

The **victim's own device** (рџ“±). The `secure_pin` requirement ensures only the physical
victim (who knows their PIN) can resolve the alert вЂ” a relay device cannot do this.

### Request Fields

| Field | Origin | How It Is Generated |
|-------|:------:|---------------------|
| `sos_id` | рџ“± Victim's device | Stored locally after SOS creation (returned by `/relay` or `/trigger`). |
| `victim_id` | рџ“± Victim's device | Loaded from `EncryptedSharedPreferences` (set at registration). |
| `secure_pin` | рџ‘¤ User input | The PIN the victim typed during this resolve action (a numeric/string PIN displayed in the app's "I'm Safe" dialog). This is compared against `profiles.secure_pin` which the user set during registration. **Not** the `emergency_token`. |

### Backend-Generated Fields (NOT in request body)

| Field | Origin | How It Is Generated |
|-------|:------:|---------------------|
| Auth check | рџ–ҐпёЏ Backend | `_verify_secure_pin(victim_id, secure_pin)` queries `profiles`. Returns 401 on mismatch. |
| Active guard | рџ–ҐпёЏ Backend | `.eq("status", "active")` clause on the `UPDATE` вЂ” if the SOS is already resolved or doesn't exist, 0 rows are affected and a 404 is returned. Prevents double-resolve. |
| `status = "resolved"` | рџ–ҐпёЏ Backend | Written to `sos_events` by the backend `UPDATE`. |
| `resolved_at` | рџ–ҐпёЏ Backend | `datetime.now(timezone.utc)` вЂ” server-side UTC timestamp. Written to `sos_events.resolved_at`. Also included in the response. |
| Safe SMS background task | рџ–ҐпёЏ Backend | `_send_safe_sms()` is enqueued as a `BackgroundTask`. Fetches contacts and sends "User is Safe" SMS via Fast2SMS. |

### Success Response Fields

| Field | Origin | Description |
|-------|:------:|-------------|
| `success` | рџ–ҐпёЏ Backend | Always `true` on 200. |
| `message` | рџ–ҐпёЏ Backend | `"SOS resolved. Contacts are being notified you are safe."` |
| `sos_id` | рџ–ҐпёЏ Backend | Echo of the resolved SOS UUID. |
| `resolved_at` | рџ–ҐпёЏ Backend | ISO 8601 UTC timestamp of resolution. The app can display this to the user as the official "resolved at" time. |

| Key | Set When | Cleared When | Used For |

| `victim_id` | After Supabase sign-up | Never (account-level) | Every API call |

| `sos_id` | After `/relay` or `/trigger` 201 response | After `/resolve` succeeds | location + resolve calls |

## Security Notes

- The **relay device never sees or stores** `secure_pin` вЂ” it only forwards the
  `SOSPacket` which contains `victim_id`, `emergency_token`, `lat`, `lng`, `emergencyType`,
  and `hopCount`. The `secure_pin` is never broadcast over Bluetooth mesh.
- The backend **ignores who the relay device is** вЂ” it trusts only `victim_id` +
  `emergency_token`. The relay's identity / IP is logged as `relay_ip` for audit, never
  for access control.
- `resolved_at` and `updated_at` are always set server-side so clients cannot spoof
  timestamps.
- `sos_id` is a server-generated UUID that cannot be guessed. A victim must receive it
  in the `/trigger` or `/relay` response before sending location updates or resolving.
